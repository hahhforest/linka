# Phase 03C UI 接真实 Room API

## 目标

UI 首屏仍展示 room workspace，但数据来源优先使用 daemon Room API。daemon 不在线或 Room API 请求失败时，UI 自动回退到 Phase 02C 的 demo room，不要求用户先手动创建 room 或发送消息。

## 实现范围

- `packages/ui/src/services/roomsService.ts` 封装 Room API：list/create/get room、list/add members、list/send messages。
- `packages/ui/src/store/roomStore.ts` 统一管理 room、members、messages、active room 和 composer 发送状态。
- `App.tsx` 启动时初始化 room store，同时保留 daemon health polling。
- `RoomWorkspace`、`RoomNav`、`Timeline`、`MemberRail`、`Composer` 均通过 store 数据渲染，不再直接读取 fixture。
- `Composer` 在 API room 可用时 POST message 并刷新历史；API 不可用时追加本地 fallback message，并保持本地暂存提示。
- 本阶段未引入 WebSocket；SSE 暂不接入，发送后刷新历史满足首轮真实 API 使用路径。

## 初始化策略

1. UI 调用 `GET /linka/rooms`。
2. 如果 daemon 返回已有 rooms，选中第一间 room，并加载 members/messages。
3. 如果 daemon 返回空列表，UI 通过 API 创建一个 demo-like room，添加 Human、LinkA、资料 Agent、核验 Agent，并发送第一条用户 instruction。
4. 任一 API 步骤失败时，store 切换到 `fallback`，展示本地 demo room。

## 验证

- `roomsService.test.ts` 覆盖各 Room API service 的 URL、method 和 body shape。
- `roomStore.test.ts` 覆盖 daemon offline 初始化回退，以及 fallback composer 本地追加。
- 保留 `demoRoom.test.ts` sanity，确保 fixture 仍完整。
